

# circuit移动端技术选型报告

### 1.背景

circuit是一个ai问答平台（RAG），也支持多模态能力。

目前已有web app，需要支持ios和安卓双端。



### 2.技术方案总览

首先，移动端存在三个大类的技术选型：web app；跨端；native

web app是指直接用web代码，进行PC和移动端的双端适配，代码跑在浏览器里面。



#### 2.1 Web App

 就是直接用前端网页技术来做，核心是 HTML、CSS、JavaScript。
 同一套代码既适配 PC，也适配手机，用户通过浏览器访问，代码运行在浏览器环境里。

#### 2.2 跨端

 跨端是指：主要写一套代码，然后同时生成或运行在 iOS 和 Android 两端。
 它不是直接跑在浏览器里，而是借助一层跨端框架，把业务代码映射到移动端能力上。

#### 2.3 Native

 Native 就是原生开发。
 iOS 用 Swift / Objective-C，Android 用 Kotlin / Java。
 代码直接跑在操作系统原生环境里，UI 组件、系统能力、性能调度，都是直接对接平台本身。



web app我们已经有了，所以要从跨端和native里面选型。



#### 2.4 三类移动端方案对比

| 维度     | Web App | 跨端 App | Native App |
| -------- | ------- | -------- | ---------- |
| 性能     | ★★☆☆☆   | ★★★★☆    | ★★★★★      |
| 开发效率 | ★★★★★   | ★★★★☆    | ★★☆☆☆      |
| 成本     | ★★★★★   | ★★★★☆    | ★★☆☆☆      |
| 人才供给 | ★★★★★   | ★★★★☆    | ★★☆☆☆      |

#### 2.5 总结

Web App 胜在快、便宜、好招人，但性能上限最低；
 Native 胜在性能最好，但开发慢、成本高、招聘也更难；
 跨端整体最平衡，性能接近原生，同时效率和成本又明显优于双原生。

结合我们的circuit业务，目前看来性能上，跨端已经足够。



### 3.技术方案横向对比

#### 3.1跨端方案

主流跨端方案：React Native、flutter、Kotlin Multiplatform  、ionic

目前已有app是基于ionic做的，app套壳，里面内嵌一个webview，然后在里面放web app。



React Native > Flutter > Ionic > Kotlin Multiplatform



| 方案                     | 性能  | 生态  | 招聘  | 维护成本 |
| ------------------------ | ----- | ----- | ----- | -------- |
| **React Native**         | ★★★★☆ | ★★★★★ | ★★★★★ | ★★★★☆    |
| **Flutter**              | ★★★★★ | ★★★★☆ | ★★★☆☆ | ★★★☆☆    |
| **Ionic**                | ★★☆☆☆ | ★★★☆☆ | ★★★★★ | ★★★★★    |
| **Kotlin Multiplatform** | ★★★★☆ | ★★☆☆☆ | ★★☆☆☆ | ★★☆☆☆    |

------

##### 总结

**React Native**
 综合最均衡，比较像“性能、生态、招人、长期可维护性”都不差的方案。对前端团队最友好，招聘池也最大。

**Flutter**
 性能最好，UI 一致性强，但团队要接受 Dart 和整套新体系，招聘难度会比 RN 高一点。

**Ionic**
 最省事，最适合已有 WebView 套壳继续迭代；但它本质还是 Web 跑在 WebView 里，性能和原生体验上限最低。

**Kotlin Multiplatform**
 更像“原生团队提效方案”，不是最通用的前端跨端方案。共享业务逻辑很强，但生态、招聘、维护门槛都更高。



**综合推荐顺序：**
**React Native > Flutter > Ionic > Kotlin Multiplatform**



#### 3.2 native方案

native的技术方案也有调研，可以作为一个参考。



ios端主要用的是swift，如果用native可能优先开发ios端。

安卓用kotlin，这个后续再看。

| 语言       | 性能  | 生态  | 招聘  | 维护成本 |
| ---------- | ----- | ----- | ----- | -------- |
| **Swift**  | ★★★★★ | ★★★★☆ | ★★★★☆ | ★★★★☆    |
| **Kotlin** | ★★★★★ | ★★★★★ | ★★★★★ | ★★★★☆    |

swift有一个优势：跨端框架也可以接近 iOS 风格，但默认组件体系不同，最终会存在一定差异；swift更容易做到与 iOS 系统原生体验完全一致；



### 4. 推荐路线

**建议采用：跨端为主，原生兜底。**

原因很简单：

- **大部分业务页面** 用跨端开发，迭代更快、复用更高
- **关键能力和性能敏感部分** 用原生承接，保证体验和稳定性

这也是目前很多 AI App 更常见的做法：
 **不是纯 WebView，也不是全原生，而是效率和体验之间取平衡。**

对 AI 问答平台来说，这种方案通常最合适：

- **聊天页、会话列表、设置页等常规页面**：适合跨端
- **推送、语音、上传、权限、分享等能力**：适合原生

落地上，可以采用 **RN + Swift/Kotlin** 的组合：
 **RN 负责主业务，iOS/Android 原生负责底层能力和关键链路。**

















# Circuit 移动端技术选型报告

## 1. 背景

Circuit 是一个 **AI 问答平台（RAG）**，支持多模态能力。

目前已有 **Web App**，需要扩展支持 **iOS 与 Android 移动端**。

移动端方案需要在 **开发效率、性能、长期维护成本**之间取得平衡。

------

# 2. 移动端技术路线

移动端开发通常有三种技术路线：

- Web App
- 跨端框架
- Native 原生开发

### 2.1 Web App

使用 **HTML / CSS / JavaScript** 开发，通过浏览器访问。

同一套代码同时适配 PC 和移动端，代码运行在浏览器环境中。

### 2.2 跨端开发

跨端框架通过 **一套代码同时支持 iOS 与 Android**。

业务代码通过框架映射到移动端能力，而不是运行在浏览器中。

### 2.3 Native 原生开发

原生开发直接运行在操作系统环境中：

- iOS：Swift / Objective-C
- Android：Kotlin / Java

UI 组件、系统能力和性能调度都直接由系统提供。

------

## 2.4 三类方案整体对比

| 维度     | Web App | 跨端 App | Native App |
| -------- | ------- | -------- | ---------- |
| 性能     | ★★☆☆☆   | ★★★★☆    | ★★★★★      |
| 开发效率 | ★★★★★   | ★★★★☆    | ★★☆☆☆      |
| 开发成本 | ★★★★★   | ★★★★☆    | ★★☆☆☆      |
| 人才供给 | ★★★★★   | ★★★★☆    | ★★☆☆☆      |

**结论**

- **Web App**：开发最快、成本最低，但性能和体验上限最低
- **Native**：性能和体验最佳，但开发成本最高
- **跨端**：效率与性能之间最均衡

对于 **AI 问答类应用**，跨端方案通常已经可以满足性能需求。

------

# 3. 技术方案对比

## 3.1 跨端框架对比

主流跨端方案：

- React Native
- Flutter
- Ionic
- Kotlin Multiplatform

目前已有 App 基于 **Ionic（WebView 套壳）**实现。

### 跨端框架对比

| 方案                 | 性能  | 生态  | 招聘  | 维护成本 |
| -------------------- | ----- | ----- | ----- | -------- |
| React Native         | ★★★★☆ | ★★★★★ | ★★★★★ | ★★★★☆    |
| Flutter              | ★★★★★ | ★★★★☆ | ★★★☆☆ | ★★★☆☆    |
| Ionic                | ★★☆☆☆ | ★★★☆☆ | ★★★★★ | ★★★★★    |
| Kotlin Multiplatform | ★★★★☆ | ★★☆☆☆ | ★★☆☆☆ | ★★☆☆☆    |

### 简要结论

**React Native**

整体最均衡的方案，在性能、生态、招聘和维护成本上都较稳定。
 对前端团队最友好，招聘池最大。

**Flutter**

性能强、UI 一致性高，但需要引入 **Dart 语言与新的技术体系**，团队学习成本更高。

**Ionic**

开发成本最低，但本质仍是 **Web 运行在 WebView 中**，性能和体验上限较低。

**Kotlin Multiplatform**

更适合 **原生团队共享业务逻辑**，但生态成熟度和招聘供给较弱。

**综合推荐顺序**

React Native > Flutter > Ionic > Kotlin Multiplatform

## 3.2 Native 原生方案

如果采用原生开发：

- iOS：Swift
- Android：Kotlin

| 技术   | 性能  | 生态  | 招聘  | 维护成本 |
| ------ | ----- | ----- | ----- | -------- |
| Swift  | ★★★★★ | ★★★★☆ | ★★★★☆ | ★★★★☆    |
| Kotlin | ★★★★★ | ★★★★★ | ★★★★★ | ★★★★☆    |

**Swift 的优势**

原生开发可以完全遵循 iOS 设计体系，在系统交互、动画和体验细节上更容易做到与系统一致。

------

# 4. 推荐方案

**推荐方案：跨端为主，原生能力补充**

原因：

1. **大部分业务页面适合跨端开发**，迭代效率更高
2. **原生能力保证系统能力和性能上限**
3. **整体成本明显低于双原生开发**

这也是目前很多 **AI App 的常见技术架构**。

------

## 技术落地方案

建议采用：

**React Native + Swift / Kotlin**

职责划分：

**跨端页面**

- 聊天页面
- 会话列表
- 设置页
- 常规业务页面

**原生能力**

- 推送
- 音视频
- 文件上传
- 权限管理
- 系统分享

**架构模式**

- RN 负责主要业务页面
- iOS / Android 原生负责底层系统能力与性能敏感模块
