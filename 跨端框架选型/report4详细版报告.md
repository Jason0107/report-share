# circuit 移动端技术选型报告

## 1. 背景

Circuit 是一个 AI 问答平台（RAG），同时支持多模态能力。
 目前已有 Web App，需要进一步支持 iOS 和 Android 双端。

本次选型重点是：在已有 Web 能力基础上，评估移动端方案的可行性与性价比，选择适合 Circuit 的落地路径。

------

## 2. 移动端技术方案总览

移动端技术方案通常可分为三类：**Web App、跨端、Native**。

### 2.1 Web App

Web App 是指直接使用前端网页技术开发，核心技术为 HTML、CSS、JavaScript。
 同一套代码适配 PC 和移动端，用户通过浏览器访问，代码运行在浏览器环境中。

### 2.2 跨端

跨端是指：主要编写一套代码，同时覆盖 iOS 和 Android 两端。
 它并不是直接运行在浏览器里，而是依赖跨端框架，将业务代码映射到移动端能力上。

常见方案包括：React Native、Flutter、Kotlin Multiplatform、Ionic 等。

### 2.3 Native

Native 即原生开发。
 iOS 主要使用 Swift / Objective-C，Android 主要使用 Kotlin / Java。
 代码直接运行在操作系统原生环境中，UI 组件、系统能力与性能调度均直接对接平台本身。

------

## 3. 三类方案对比

由于 Circuit 已有 Web App，Web 方案已具备基础能力；本次重点是在**跨端**与**Native**之间评估主路线，同时将 Web App 作为对照参考。

### 3.1 三类移动端方案对比

| 维度     | Web App | 跨端 App | Native App |
| -------- | ------- | -------- | ---------- |
| 性能     | ★★☆☆☆   | ★★★★☆    | ★★★★★      |
| 开发效率 | ★★★★★   | ★★★★☆    | ★★☆☆☆      |
| 成本     | ★★★★★   | ★★★★☆    | ★★☆☆☆      |
| 人才供给 | ★★★★★   | ★★★★☆    | ★★☆☆☆      |

### 3.2 小结

Web App 的优势在于开发快、成本低、复用现有 Web 能力强，但性能与系统能力上限最低。
 Native 的优势在于性能、体验、系统能力最强，但开发成本最高，通常需要分别维护 iOS 与 Android 两套能力。
 跨端整体处于两者之间，在性能、效率、成本之间取得了较好的平衡。

结合 Circuit 的业务特点来看，AI 问答、多模态上传播放这类场景，对极致图形性能要求并不高，**跨端方案通常已经能够满足核心业务需求**。

------

## 4. 跨端方案横向对比

当前主流跨端方案包括：**React Native、Flutter、Ionic、Kotlin Multiplatform**。

其中，当前已有 App 方案是基于 **Ionic**：
 即 App 套壳，内部通过 WebView 承载 Web App。

### 4.1 综合排序

**React Native > Flutter > Ionic > Kotlin Multiplatform**

### 4.2 方案对比

| 方案                     | 性能  | 生态  | 招聘  | 维护成本 |
| ------------------------ | ----- | ----- | ----- | -------- |
| **React Native**         | ★★★★☆ | ★★★★★ | ★★★★★ | ★★★★☆    |
| **Flutter**              | ★★★★★ | ★★★★☆ | ★★★☆☆ | ★★★☆☆    |
| **Ionic**                | ★★☆☆☆ | ★★★☆☆ | ★★★★★ | ★★★★★    |
| **Kotlin Multiplatform** | ★★★★☆ | ★★☆☆☆ | ★★☆☆☆ | ★★☆☆☆    |

### 4.3 各方案简述

**React Native**
 综合最均衡。性能、生态、招聘、长期可维护性都较好，对前端团队最友好，迁移成本也相对较低。

**Flutter**
 性能强，UI 一致性高，但需要引入 Dart 和整套 Flutter 技术体系，团队学习成本和招聘门槛高于 RN。

**Ionic**
 最适合已有 WebView 套壳继续迭代，复用现有 Web 资产最多；但本质仍是 Web 跑在 WebView 中，性能和原生体验上限最低。

**Kotlin Multiplatform**
 更适合原生团队做逻辑共享，不是典型意义上的“前端跨端 UI 方案”。对当前团队结构而言，门槛较高，短期性价比不占优。

### 4.4 小结

从 Circuit 当前团队能力、已有 Web 基础、未来双端建设成本来看，**React Native 是当前最均衡的跨端主选项**。
 Flutter 更适合愿意重建技术体系、追求更强 UI 控制力的团队；Ionic 更适合短期低成本延续；Kotlin Multiplatform 暂不建议作为主方案。

------

## 5. Native 方案补充说明

如果选择 Native 路线，iOS 侧主要采用 **Swift**，Android 侧主要采用 **Kotlin**。
 从实际落地看，如果资源有限，通常会优先建设 iOS 端，再评估 Android 端的投入节奏。

Swift 的优势在于：
 即使跨端框架可以尽量接近 iOS 风格，但默认组件体系、交互细节、系统一致性上，仍会与纯原生存在差异；而 Swift 更容易做到与 iOS 系统体验完全一致。

因此，Native 更适合承担以下能力：

- App 壳层
- 登录与启动链路
- 推送
- 音频录入与播放
- 图片 / 视频选择与上传
- 权限管理
- 性能监控与崩溃治理
- 系统级分享、剪贴板、文件能力等

------

## 6. 推荐路线

### 6.1 推荐主路径

**推荐路线：混合开发，跨端为主，原生兜底。**

即：

- 业务主界面与主要交互使用跨端方案承载
- 壳层与关键系统能力由原生承接
- 避免继续走纯 WebView 套壳的长期路线

### 6.2 推荐原因

参考豆包、元宝等头部 AI App 的常见做法，行业主流并不是纯 WebView 路线，而是：

**保留原生客户端能力 + 关键链路原生兜底 + 业务层通过跨端提升迭代效率**

对于 Circuit 这类 AI 问答平台而言，这条路线通常具备更高性价比：

- 比纯 Native 更快
- 比纯 WebView 体验更好
- 比双原生更省成本
- 比继续深度依赖 Ionic 更具长期演进空间

### 6.3 推荐结论

**主推荐方案：React Native + 原生能力兜底**

不建议继续将 Ionic / WebView 作为长期主路线；
 Native 建议作为壳层和关键能力补充，而不是整体双原生重做。

------

## 7. 最终结论

综合业务场景、现有技术基础、开发效率、成本与长期演进空间，Circuit 更适合采用：

**“React Native 为主，原生兜底”的混合开发路线。**

该方案能够在性能、体验、开发效率与团队可承接性之间取得较好的平衡，适合作为 Circuit 移动端建设的主路径。
