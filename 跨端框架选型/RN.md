

# atomsphere，它会支持RN吗

公开信息有限，但根据搜索结果可以看到：

Atmosphere 目前大概率不支持 React Native。

它是 Cisco 内部的设计系统，和 Muse（用于 cisco.com）、Momentum（用于 Webex 等产品）并列。从公开资料看，Atmosphere 主要聚焦在 Web 端组件（设计 token、颜色体系、页面主题等），没有看到 React Native 的适配。

Cisco 内部几套设计系统的定位：

| 设计系统   | 用途                 | RN 支持                  |
| :--------- | :------------------- | :----------------------- |
| Momentum   | Webex 等产品 UI      | 有 Web Components，无 RN |
| Muse       | cisco.com 营销页面   | 纯 Web                   |
| Atmosphere | 内部产品（你们用的） | 大概率纯 Web             |

我的建议：

1. 先跟 Atmosphere 团队确认一下他们有没有 RN 支持的计划或者 roadmap

1. 如果没有，可以做的是：把 Atmosphere 的设计 token（颜色、字号、间距、圆角）提取出来，在 RN 里用 constants/theme.ts 对齐，确保视觉一致性

1. 组件层面自己用 RN 基础组件实现，或者引入 Tamagui/NativeWind 按 Atmosphere 的规范来定制

这也是很多公司的做法——设计系统出规范和 token，各端各自实现组件，保持视觉统一就行。



# 这个rn和另外一个团队用react写的，放到手机浏览器里面，有啥区别？

好问题，这是选型时经常被问到的。核心区别：

## React（Web）放手机浏览器 vs React Native（原生 App）

| 维度     | Web 端（手机浏览器打开）                   | React Native（原生 App）               |
| :------- | :----------------------------------------- | :------------------------------------- |
| 渲染引擎 | 浏览器渲染 HTML/CSS/DOM                    | 直接调用 iOS/Android 原生 UI 组件      |
| 性能     | 受浏览器沙箱限制，复杂列表和动画容易卡     | 接近原生性能，长列表和动画流畅         |
| 体验感   | 像网页，滑动有橡皮筋但不够丝滑             | 像微信/抖音那种原生手感                |
| 入口     | 要打开浏览器、输 URL 或点书签              | 桌面有 App 图标，点开即用              |
| 离线能力 | 基本不行（PWA 有限支持）                   | 可以离线使用                           |
| 推送通知 | 浏览器通知，体验差且常被拦截               | 系统级推送，和微信一样                 |
| 硬件访问 | 有限（相机、GPS 能用，但要授权且体验粗糙） | 完整（相机、生物识别、NFC、蓝牙等）    |
| 手势交互 | 只有基本点击和滚动                         | 左滑返回、下拉刷新、复杂手势都原生支持 |
| 发布方式 | 部署到服务器就行                           | 需要上架 App Store / Google Play       |

## 对你们这个 AI 问答场景来说

Web 端放浏览器其实够用——输入框、聊天列表、Markdown 渲染，浏览器都能做。但做原生 App 的优势在于：

1. 用户留存 — 桌面有图标，打开率远高于浏览器书签

1. 推送召回 — "你的报告分析好了" 这种通知只有 App 能做好

1. 体验质感 — 原生的键盘弹起、手势返回、页面过渡动画，是 Web 模拟不出来的

1. 企业内部分发 — 公司 MDM 可以直接推 App 到员工手机

简单说：Web 是"能用"，App 是"好用"。 你们做移动端 App 就是为了把体验从"能用"提升到"好用"。



# 那豆包、元宝这种为啥搞了个app

和上面说的一样，核心还是体验和留存。但对 AI 助手类产品来说，还有几个特别关键的原因：

## 1. 语音交互是刚需

豆包主打语音对话，需要：

- 常驻后台录音

- 语音唤醒（"你好豆包"）

- 实时语音流式传输

这些浏览器根本做不了，或者做了体验很烂。

## 2. 多模态能力

- 拍照识题 — 直接调原生相机，拍完即识别

- 实时翻译 — 摄像头取景框里实时 AR 翻译

- 文件/图片处理 — 从相册选、从其他 App 分享过来

浏览器虽然也能调相机，但启动慢、没有自定义取景框、不能后台处理。

## 3. 抢占用户桌面

AI 助手是高频工具，每天用好多次。桌面有图标 vs 打开浏览器输网址，打开率差 10 倍不止。App 图标就是品牌曝光。

## 4. 推送驱动活跃

- "你昨天问的问题有更新了"

- "今日热门话题推荐"

- "你的文档生成好了"

系统级推送是拉活最有效的手段，Web 通知基本没人开。

## 5. 商业变现

App Store 内购、会员订阅，苹果/谷歌有成熟的支付体系。Web 支付要自己对接，还容易被拦截。

## 总结

浏览器里的 AI 对话适合轻度尝鲜，但要做成日常工具让用户养成习惯，必须是 App。豆包、元宝、Kimi 这些全都是 Web + App 双端，Web 负责拉新转化，App 负责留存和深度使用。

你们做的 CIRCUIT 也是同理——Web 端已经有了，现在做 App 就是让员工从"偶尔用用"变成"天天用"





# 你觉得是RN直接做三端好，还是web做一个端，rn做两个端好

对于你们这个场景（企业内部 AI 助手），我建议 Web 一端 + RN 两端，也就是你们现在的方案。原因：

## RN 三端统一的理想很美好，但现实有坑

听起来好的地方：

- 一套代码三端跑，人力省

- 逻辑不用写两遍

实际踩的坑：

- Web 端用 react-native-web 写出来的页面，和用 React 写的 Web 体验差距很大 — 没有 SEO、没有原生 HTML 语义、没有浏览器原生交互习惯（右键、hover、快捷键）

- 样式要不停写 Platform.OS === 'web' 的判断，代码越来越脏

- Web 端的响应式布局（大屏、中屏、小屏）用 RN 做非常别扭

- 生态不同：Web 有大量成熟库（表格、图表、富文本编辑器），RN 对应的库少且质量参差不齐

## 为什么分开更好

|          | Web 团队（React）          | 移动端团队（RN）         |
| :------- | :------------------------- | :----------------------- |
| 目标用户 | 坐在电脑前办公的人         | 移动办公、会议间隙用手机 |
| 交互模式 | 键鼠、大屏、多窗口         | 触摸、小屏、单任务       |
| 功能侧重 | 复杂编辑、文件管理、多 tab | 快速问答、语音输入、推送 |
| 迭代节奏 | Web 部署秒生效             | App 要审核发版           |

两端的用户场景、交互方式、功能优先级都不一样，强行共用一套代码反而限制了各自做到最好。

## 真正该共用的是什么

不是 UI 代码，而是：

- API 层 — 后端接口一套，两端各自调

- 数据模型/类型定义 — 可以抽 npm 包共享 TypeScript 类型

- 业务逻辑 — 比如消息解析、Markdown 处理，可以抽纯 JS 的 shared 包

- 设计规范 — 设计 token（颜色、字号）统一，组件各端各自实现

这也是大厂的主流做法 — **豆包、Kimi、ChatGPT 都是 Web 独立做，App 独立做，共享后端和设计规范**。





# 大厂的app独立做，是ios和安卓同构吗

要分情况，两种路线都有：

## 路线一：RN/Flutter 跨端同构（iOS + Android 一套代码）

代表产品：

- Discord — React Native

- Shopify — React Native

- Meta 系（Facebook、Instagram 部分页面）— React Native

- 闲鱼 — Flutter

- 字节系（头条、豆包部分）— Flutter

优点： 一套代码出两端，人力省一半

缺点： 极端性能场景（复杂动画、视频编辑）有瓶颈

## 路线二：iOS/Android 各自原生（Swift + Kotlin）

代表产品：

- 微信 — 各端原生

- 支付宝 — 各端原生

- 抖音核心体验 — 各端原生

- ChatGPT App — Swift（iOS）+ Kotlin（Android）

优点： 性能极致，平台特性完全利用

缺点： 需要两个团队，成本翻倍

## 现在的趋势

大部分公司的做法是 混合策略：

- 核心高性能页面（视频播放、复杂手势）用原生写

- 业务页面（设置、列表、表单、聊天）用跨端框架写

比如抖音：视频流用原生，电商/直播间用 Flutter/Lynx。

## 对你们来说

AI 问答这个场景 — 输入框、消息列表、Markdown 渲染 — 完全是跨端框架的舒适区，不涉及高性能原生需求。用 RN 一套代码出 iOS + Android 就是最合理的选择，没必要两端各写一遍。



# 这两者如何结合？

“核心高性能页面（视频播放、复杂手势）用原生写

业务页面（设置、列表、表单、聊天）用跨端框架写”

这两者如何结合？



核心思路就是 跨端框架做主壳，原生做特定模块，通过"桥"通信。几种常见的结合方式：

## 1. Native Module（原生模块）

最常用。RN 壳里调用原生代码。

┌─────────────────────────────┐

│   RN 层（JS/TS）      │

│ 聊天页、设置页、列表页    │

│     │          │

│  Bridge / JSI       │

│     │          │

│ ┌──────┴──────┐      │

│ │ Native Module│      │

│ │ 视频播放器  │      │

│ │ 人脸识别   │      │

│ └─────────────┘      │

└─────────────────────────────┘

比如你要加一个高性能视频播放器：

- iOS 端用 Swift 写 AVPlayer 封装

- Android 端用 Kotlin 写 ExoPlayer 封装

- RN 层通过 NativeModule 调用，JS 只管传参和监听事件

## 2. Native View（原生视图嵌入）

把一个原生写的 View 嵌到 RN 页面里，就像用一个普通组件一样。

// RN 页面里直接用原生视频组件

<View>

 <Text>视频标题</Text>

 <NativeVideoPlayer videoUrl="..." /> // 这个是原生渲染的

 <Text>评论区</Text>          // 这些是 RN 渲染的

</View>

同一个页面里，视频部分原生渲染保证性能，其他部分 RN 渲染保证开发效率。

## 3. 原生页面跳转

整个页面是原生写的，RN 通过导航跳过去。

RN 首页 → RN 聊天页 → 原生视频编辑页 → 返回 RN

用户感知不到切换，但背后两套渲染引擎各干各的。

## 实际举个例子

假设你们 CIRCUIT 后面要加 视频会议 功能：

| 页面     | 用什么写 | 原因                     |
| :------- | :------- | :----------------------- |
| AI 聊天  | RN       | 纯业务逻辑，跨端效率高   |
| 历史记录 | RN       | 列表页，RN 完全胜任      |
| 设置页   | RN       | 表单页，没必要原生       |
| 视频通话 | 原生模块 | 实时音视频对性能要求极高 |
| 相机扫码 | 原生模块 | 需要直接操控摄像头硬件   |

RN 负责 80% 的业务页面，原生补齐 20% 的性能关键模块。 这就是混合架构的核心理念——不是二选一，而是各取所长。

对你们 POC 阶段来说，100% RN 就够了，这些后面需要的时候再加不迟。
